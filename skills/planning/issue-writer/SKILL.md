# issue-writer

한 줄 설명: 흩어진 요구사항을 GitHub, Linear 등 여러 이슈 트래커에서 바로 쓸 수 있는 실행 가능한 이슈 초안으로 바꾸는 스킬이다.

## 언제 사용해야 하는지

- PRD나 요구사항 문서를 실행 가능한 작업 이슈로 분해해야 할 때
- API 계약 변경사항을 백엔드 또는 프론트엔드 작업 이슈로 등록해야 할 때
- 회의 메모, 슬랙 요청, 버그 제보를 정리된 실행 항목으로 바꿔야 할 때
- Linear, GitHub, 기타 이슈 트래커에 붙여 넣을 수 있는 범용 이슈 설명이 필요할 때
- 완료 조건과 범위 밖 항목을 미리 명확히 적어 리뷰 비용을 줄이고 싶을 때

## 입력으로 기대하는 정보

### 필수

- 작업 제목 또는 해결하려는 문제
- 배경 또는 요청 맥락
- 기대 결과 또는 완료 상태

### 권장

- 사용할 이슈 트래커 종류 (`Linear`, `GitHub`, 기타)
- 관련 PRD, API 문서, 디자인 링크, 회의 메모
- 작업 범위와 제외 범위
- 담당 팀 또는 예상 협업 대상
- 일정, 우선순위, 의존 작업
- 라벨, 프로젝트, 팀, 우선순위 같은 메타데이터

### 입력이 부족할 때 처리 규칙

- 완료 조건이 불명확하면 검증 가능한 상태 기준으로 재정리한다.
- 범위가 넓으면 한 이슈에 모두 넣지 말고 현재 이슈 범위와 후속 이슈 후보를 분리한다.
- 참고 링크가 없으면 `참고 사항`에 필요한 추가 자료를 적는다.
- 이슈 트래커가 지정되지 않으면 Linear와 GitHub에 모두 붙여 넣기 쉬운 범용 마크다운 형식으로 작성한다.

## 반드시 포함해야 하는 출력 형식

항상 아래 범용 이슈 템플릿 형태의 마크다운으로 작성한다.

```md
# [Issue Title]

## 메타데이터 (선택)
- 트래커: Linear / GitHub / 기타
- 팀:
- 우선순위:
- 라벨 또는 프로젝트:
- 의존 작업:

## 배경
- ...

## 문제 또는 목표
- ...

## 해야 할 일
- [ ] ...
- [ ] ...

## 완료 조건
- [ ] ...
- [ ] ...

## 참고 사항
- 관련 문서:
- 협업 필요 팀:
- 참고 링크:

## 범위 밖 항목
- ...
```

### 출력 규칙

- 제목은 결과물이 보이도록 동사형보다 작업 단위 중심으로 쓴다.
- `메타데이터`는 입력으로 받은 정보가 있을 때만 채우고, 없으면 비워 두거나 생략할 수 있다.
- `해야 할 일`은 구현, 문서, 테스트, 커뮤니케이션 작업을 구분 없이 실제 행동 단위로 적는다.
- `완료 조건`은 검토자가 체크 가능한 상태로 작성한다.
- `참고 사항`에는 링크가 없더라도 필요한 참고 정보의 종류를 적는다.
- `범위 밖 항목`은 이번 이슈에서 다루지 않을 내용을 명시한다.
- 트래커가 `Linear`로 명시되면 제목과 본문은 더 짧고 실행 중심으로 정리한다.
- 트래커가 `GitHub`로 명시되면 설명과 체크리스트를 조금 더 자세히 작성할 수 있다.
- 트래커가 지정되지 않으면 특정 도구 전용 문법 없이 범용 마크다운으로 작성한다.

## 작업 절차

1. 입력 내용을 읽고 해결하려는 핵심 문제 또는 목표를 한 문장으로 정리한다.
2. 왜 이 이슈가 필요한지 `배경`에 적고, 이번 범위를 벗어나는 내용은 분리한다.
3. 실제 작업자가 순서대로 처리할 수 있도록 `해야 할 일`을 액션 단위로 쪼갠다.
4. 완료 여부를 판단할 수 있도록 `완료 조건`을 작성한다.
5. 참고 문서, 협업 대상, 선행 작업이 있으면 `참고 사항`과 `메타데이터`에 정리한다.
6. 사용 트래커가 명시되면 분량과 표현을 그 도구 사용 방식에 맞게 조정한다.
7. 작업 범위가 과도하게 넓으면 후속 이슈로 분리할 항목을 `범위 밖 항목`에 적는다.

## 품질 체크리스트

- 제목만 보고도 어떤 작업인지 이해할 수 있는가
- 배경과 목표가 중복되지 않고 구분되어 있는가
- 해야 할 일이 실제 작업 단위로 쪼개져 있는가
- 완료 조건이 객관적으로 확인 가능한가
- 참고 사항에 필요한 협업 정보가 빠지지 않았는가
- 범위 밖 항목이 명확해서 불필요한 확장을 막을 수 있는가
- 이슈 하나로 처리하기 어려운 내용이 섞여 있지 않은가
- 특정 도구 전용 표현 때문에 다른 트래커로 옮기기 어렵지 않은가

## 예시 요청

- `회원 가입 PRD를 바탕으로 프론트엔드 구현용 Linear 이슈 초안을 작성해줘.`
- `결제 API 에러 스펙 변경사항을 기준으로 백엔드 대응 이슈를 써줘. 완료 조건도 포함해줘.`
- `운영팀이 요청한 주문 검색 개선 요구사항을 범용 이슈 템플릿으로 정리해줘. 범위 밖 항목도 꼭 적어줘.`
- `아래 회의 메모를 바탕으로 QA 테스트 케이스 정리 이슈 초안을 만들어줘. GitHub에도 붙여 넣을 수 있게 해줘.`
